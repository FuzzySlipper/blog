# Building a Den-Like System with Stock Hermes Components

**Date:** 2026-05-26  
**Research Question:** What stock Hermes Agent components and established extensions can approximate a Den-like system (task/project management, multi-agent orchestration, document store, visual management) for a new user, without embedding into someone else's workflow specifics?

---

## Executive Summary

Hermes Agent has mature built-in components that cover **~80% of Den's functionality** out of the box, particularly around multi-agent task management. The largest gap is a **shared project document store** (Den's docs-equivalent) — there's no built-in "project knowledge base" where agents and humans collaboratively create/read/update/search documents like PRDs, ADRs, specs, and notes. However, this gap is easily bridged with MCP servers for local markdown files, Obsidian vaults, or a lightweight custom MCP document server.

The recommended stack for a friend:

| Den Component | Hermes Equivalent | Status |
|---|---|---|
| Task management | Kanban (built-in) | ✅ First-class |
| Multi-agent orchestration | Kanban dispatcher + agents | ✅ First-class |
| Project-level separation | Multiple Kanban boards / tenants | ✅ Built-in |
| Visual frontend | Hermes Workspace or built-in Dashboard | ✅ Community / Built-in |
| Knowledge / document store | MCP filesystem server + qmd search OR Obsidian MCP | 🟡 Needs config |
| Agent memory | Built-in MEMORY.md / memory providers | ✅ First-class |

## 1. Kanban — Project & Task Management

### What it is

Kanban is a **durable, SQLite-backed task board** shared across all Hermes profiles on a host. It was introduced in Hermes v0.12.0 as the primary multi-agent collaboration primitive.

### Key Capabilities Relevant to "Den-like PM"

| Capability | Details |
|---|---|
| **Multiple boards** | The `--tenant` flag provides data isolation within a board. Recent updates also support true **multiple boards** for cleaner project separation. |
| **Task hierarchy** | Parent-child dependencies with automatic structured handoff (parent summary + metadata passed to child workers). |
| **6-column workflow** | Triage → Todo → Ready → In Progress → Blocked → Done |
| **Auto-decomposition** | Put a goal in Triage; the decomposer auto-fan-outs to specialist profiles based on their descriptions. |
| **Retry & circuit breakers** | Configurable failure limit (default 2), auto-blocks on repeated failures. |
| **Crash recovery** | Dispatcher monitors worker PIDs; releases claim on crash so a fresh worker can retry. |
| **Structured handoff** | `kanban_complete(summary=..., metadata={...})` carries changed files, test results, residual risk between pipeline stages. |
| **Run history** | Every attempt is a durable row — no lost context on retry. |
| **Profile isolation** | Each profile has its own `kanban-worker`/`kanban-orchestrator` skills, memory, and tools. |
| **Gateway integration** | `/kanban` slash commands in Telegram/Discord/Slack for human-in-the-loop management. |

### Does it do "project management"?

**Yes.** Multiple boards → projects. Task hierarchies with dependencies → project plans. Role-based dispatch → team workflows. The missing piece compared to Den is a dedicated document store attached to each project/board — Kanban tasks have a body/comments field but not a separate document namespace.

### What Kanban doesn't do

- No per-project document namespace (docs live separately)
- Single-host only (no multi-machine coordination)
- Manual dispatcher setup needed (gateway-embedded dispatcher in recent versions)

### Configuration

```yaml
# ~/.hermes/config.yaml
kanban:
dispatch_interval_seconds: 60
failure_limit: 2
auto_decompose: true
```

Profile skills:
```
hermes skill install kanban-worker   # for worker profiles
hermes skill install kanban-orchestrator  # for manager profiles
```

## 2. Visual Frontend Options

### A. Built-in Dashboard (Port 9119)

Ships with Hermes. Provides drag-and-drop Kanban board with WebSocket live updates. Drawer view for editing task descriptions, managing dependencies, and viewing run history. Binds to localhost by default.

**Command:** `hermes dashboard`

**Limitations:** Chat-focused; no terminal, file browser, or document management.

### B. Hermes Workspace (⭐ 2k, TS/React)

**GitHub:** https://github.com/outsourc-e/hermes-workspace

The most feature-rich frontend for "Den-like" operations:

| Feature | Details |
|---|---|
| **Chat** | Full Hermes chat with agent interaction |
| **Embedded Terminal** | xterm.js PTY |
| **Conductor** | Multi-agent task decomposition orchestrator |
| **Skills Hub** | Browse and install skills from UI |
| **Swarm Mode** | Control plane for unlimited agents with role-based dispatch lanes (Research, QA, Triage, etc.) and review gates |
| **Memory/Skills Editor** | Edit agent memory and browse skills |
| **MCP Page** | MCP catalog and marketplace |
| **Zero-fork** | Runs on vanilla `NousResearch/hermes-agent` |

**Setup options:**
- Docker Compose (~2 min): `docker compose up -d`
- One-line install: `curl -fsSL https://raw.githubusercontent.com/outsourc-e/hermes-workspace/main/install.sh | bash`
- Attach mode for existing Hermes: `npx hermes-workspace`

**Ports:** Gateway on 8642, Dashboard on 9119, Workspace on 3000

**Limitations:** Still doesn't include a dedicated project document store — that's an add-on.

### C. Other Frontends (for reference)

| Project | Stars | Best For |
|---|---|---|
| Hermes WebUI | ⭐ 3.1k | Mobile-friendly chat |
| Hermes Web UI (EKKO) | ⭐ 1.5k | Chinese platforms |
| Claw Admin | ⭐ 577 | Dual gateway |
| Hermes Control Interface | ⭐ 450 | Security/RBAC |
| Scarf (macOS) | ⭐ 239 | Native Mac app |

All can coexist on different ports connecting to the same Hermes gateway.

## 3. The Document Store Gap (Den Docs Equivalent)

### What Den provides

Den's document system is a first-class, project-scoped, versioned document store accessible via MCP tools + a librarian agent. Documents have types (PRD, ADR, spec, etc.), tags, slugs, full-text search, and agent guidance registration.

### What Hermes has for this (none is a drop-in replacement)

| Option | What It Does | Gap |
|---|---|---|
| **Built-in MEMORY.md/USER.md** | Agent-scoped, 2.2K/1.4K char limits | Per-agent, not shared project docs |
| **8 memory providers** (Honcho, OpenViking, Mem0, Hindsight, etc.) | Semantic long-term memory for agents | Agent memory, not shared document store |
| **MCP filesystem server** (`@modelcontextprotocol/server-filesystem`) | Reads/writes local files via MCP | No search, no doc types, no metadata |
| **qmd skill** (built-in optional skill) | BM25 + vector + reranking hybrid search over local markdown files | Search only; no create/update/type system |
| **LLM Wiki skill** (built-in optional skill) | Karpathy-style interlinked markdown KB | Manual maintenance; no structured doc types |
| **Obsidian MCP** (community) | Full Obsidian vault access via MCP | Requires Obsidian setup; overkill if just want project docs |
| **library-mcp** (`lethain/library-mcp`) | MCP server for markdown knowledge base folders | Basic CRUD on markdown files |
| **Docs MCP Server** (`arabold/docs-mcp-server`) | Fetches + indexes documentation from URLs/GitHub/local | Read-only, docs-focused |
| **github-knowledge-base-mcp** | Read markdown/notebooks from public GitHub repos | Read-only, GitHub-only |

### Recommended Document Store Approach

**For simplest setup:** MCP filesystem server pointing at a shared `~/projects/<board-name>/docs/` directory + the `qmd` skill for search. This gives agents the ability to read/write markdown files and search them.

```yaml
mcp_servers:
project-docs:
command: "npx"
args: ["-y", "@modelcontextprotocol/server-filesystem", 
"/home/user/projects/my-project/docs"]
```

Then load the qmd skill for search: `hermes skill install qmd`

**For structured docs (closer to Den):** Build a lightweight MCP server (or use `lethain/library-mcp`) that provides:
- `list_documents` → list docs in a project
- `read_document` → get doc content
- `write_document` → create/update with type+tags+slug
- `search_documents` → full-text search

This is the one component that would need ~100 lines of custom Python. It's the clearest "extend in a few places" point.

**For maximum polish:** Obsidian vault as MCP (structured markdown with links/graph) + search. Obsidian is free and the MCP integration is well-documented.

## 4. Putting It Together: Recommended Stack

### Tier 1: Minimal Setup (~10 min)

```
Hermes Agent (vanilla install)
├── Kanban (built-in)
│   ├── Multiple boards for project separation
│   └── Gateway-embedded dispatcher
├── Hermes Dashboard (built-in, port 9119)
│   └── Drag-and-drop Kanban visual
├── MCP filesystem server
│   └── Shared markdown docs directory
└── qmd skill (built-in optional skill)
└── Hybrid search over docs
```

### Tier 2: Full Workspace (~15 min)

```
Hermes Agent (vanilla install)
├── Kanban (built-in)
├── Hermes Workspace (port 3000)
│   ├── Chat + Terminal + Skills Hub
│   └── Conductor / Swarm Mode for multi-agent
├── MCP filesystem server + qmd
└── Or: library-mcp for structured doc operations
```

### Tier 3: Document-Forward (~20 min)

```
Hermes Agent (vanilla install)
├── Kanban (built-in)
├── Hermes Workspace (port 3000)
├── Obsidian vault + Obsidian MCP server
│   └── Structured markdown with links, tags, graph
└── qmd skill for hybrid search over vault
```

## 5. What Needs Custom Extension

1. **Document store MCP server** — ~100 lines Python to create a `project-docs` MCP server with create/read/update/search operations. Or use `lethain/library-mcp` as a starting point and add the missing CRUD operations.

2. **Project initialization convention** — A simple script or skill that creates the directory structure:
```
~/projects/<project-name>/
├── kanban/          # (auto-managed by Hermes)
├── docs/
│   ├── prd/
│   ├── spec/
│   ├── adr/
│   └── reference/
└── code/            # workspace for Kanban workers
```

3. **Optional: agent guidance / librarian equivalent** — A skill that teaches agents to consult docs before starting tasks (analogous to Den's agent guidance + librarian). This is a natural fit for the `kanban-orchestrator` skill customization.

## 6. Open Questions

- Hermes Kanban's multiple board feature: is `--tenant` sufficient, or do true separate boards now exist? (Recent YouTube demos suggest full separate boards, worth verifying against current docs.)
- Can the MCP filesystem server handle concurrent writes from multiple agent workers safely? For shared docs, some form of locking or append-only convention may be needed.
- For document-first teams, would a lightweight CMS (like Docsify, Docusaurus, or Starlight) + MCP server be better than raw markdown files?
