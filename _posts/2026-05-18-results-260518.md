## Round 4: thinking-level ablation — 11 reasoning models × weighted-split × {minimal, off}

Re-run of weighted-split only for all 11 lemonade-lan models with `thinking: yes` in pi's model config. Three-way comparison: Round 3 default (model chooses), `--thinking minimal`, `--thinking off`. Non-reasoning models (Qwen3-Coder-30B, Qwen3-Next-80B, Qwen3.5-35B) skipped — `--thinking` flag has no effect on them.

Hypothesis: for bounded, tightly-scoped tasks under orchestrated conditions, local model thinking is overhead that can hurt as much as it helps ("outsourced thinking" — the supervisor has already done the reasoning; the local model just needs to execute).

### Round 4 observations

- **Thinking was masking capability in two models.** Gemma 26B Q4 went 5/9 → 9/9 with both minimal and off, despite generating 27KB of thinking in the default run that correctly analysed the Hamilton method but failed to translate it into code. GLM-4.7 Q4 went 7/9 → 9/9. In both cases the model *can* implement the algorithm — the default thinking was interfering, not helping.

- **MiniMax Q2 is a genuine capability ceiling.** 5/9 in all three conditions. Not thinking-confused; just not capable of this task at Q2 quantisation.

- **The optimal thinking level is model-dependent, not monotone.**
- *Qwen3.6 27B Q8*: default=timeout, minimal=9/9 (797s — barely), off=timeout. Minimal is the sweet spot; too little structure and it stalls, too much and it can't finish in 900s on this hardware.
- *Gemma 31B Q4*: default=timeout, minimal=timeout, off=9/9. Needs thinking fully disabled to converge.
- *Gemma 31B Q6*: minimal and off both work; default times out.
- *Qwen3.6 27B Q4*: default=9/9, minimal=timeout, off=9/9. Minimal disrupts this model specifically — it enters the same analysis-paralysis pattern seen in its Q8 sibling, but only at minimal, not default or off.

- **Stable models are unaffected.** Gemma 26B Q6, Qwen3.6 35B Q4/Q8, GLM-4.7 Q8 all score 9/9 regardless of thinking level. These are the "just works" candidates for crew use regardless of orchestration thinking policy.

- **Wall-time improvements from reduced thinking are real but modest for passing models.** GLM Q8: 474s → 324s at minimal. Qwen3.6 35B Q8: 183s → 113s at minimal. Qwen3.6 35B Q4 essentially unchanged. The hardware bottleneck is inference speed, not thinking token count, for models that weren't timeout-bound.

- **Architecture recommendation from this data:** Run local crew models with `--thinking minimal` as the default. It unlocks Gemma 26B Q4, Gemma 31B Q6, and (just) Qwen3.6 27B Q8 without hurting the stable models. For Gemma 31B Q4 specifically, `--thinking off` is needed. A per-model thinking policy in the conductor (keyed to model ID) is the cleanest solution for a production crew setup.

### Round 3 observations

- **Roman numerals is a solved problem for every local model.** Every model — including the Q2 MiniMax and the models that couldn't implement weighted-split Hamilton — produced a correct 13/13 RN implementation. The algorithm is simple enough (greedy subtraction) that even low-quant inference gets it right. This floor tells us the test is too easy for RN on its own, but it's still useful as a style probe.
- **Weighted-split separates the field.** 7 of 14 models scored 9/9, 2 scored 7/9 or 5/9 on correctness, 4 hit the 5/9 "didn't implement it" floor (baseline retained), 1 timed out without editing, and 1 had a build error. That's real spread — unlike the cloud round sweep.
- **Quant degradation is real but model-specific.** Gemma 4 26B: Q4 fails Hamilton (5/9), Q6 passes (9/9) — clean degradation signal. Qwen 3.6 27B: Q4 passes (9/9), Q8 hits analysis paralysis (5/9 via timeout) — the Q8 is slower to infer, so the 900s wall hits before it can converge. Qwen 3.6 35B: both Q4 and Q8 pass (9/9) — the larger model holds quality across both quants. The degradation isn't monotone in quant level; it interacts with model size and the 900s cap.
- **The 900s timeout is a meaningful constraint here.** Cloud models finished in 19-685s (outlier) with a median around 80s. Local models are slower to infer; the weighted-split case (harder algorithm) took up to 474s for a 7B-class model. Three models hit the cap without converging. Raising the timeout or splitting the prompt into smaller steps may help borderline cases.
- **Qwen3-Next 80B: right idea, trivial bug, wrong case.** On WS it had the correct Hamilton structure but introduced a `long` → `int` type error on `.Take()`. On RN it produced the most elaborate control-flow (grouped if/else-if + while) and solved it in 29s. The 80B doesn't show consistent advantages over the 35B on these bounded tasks.
- **Style on RN: most local models chose table-driven.** 7 of 14 table-driven, 4 verbose (unrolled/docstring/grouped-if), 3 balanced. The verbose-hint engagement rate is similar to cloud (2/7 cloud models engaged; ~4/14 local). GLM-4.7 Q4 and Qwen3 Next 80B produced the most structurally verbose implementations. MiniMax Q2 and Qwen 3.5 35B added XML docstrings. Qwen3-Coder 30B chose a semantically motivated if/while split. The Q4 and Q8 of the same base model sometimes chose different styles (GLM-4.7 Q4=verbose, Q8=terse; Gemma 31B Q4=parallel arrays, Q6=tuple table).
- **Qwen3-Coder 30B is the standout local model.** Fastest correct WS solve (124s), fastest RN solve (25s), clean Hamilton implementation, restrained — except for the out-of-scope test file modification. That one flag keeps it from being a clear recommendation.
- **Architecture implication.** Several models that failed WS (MiniMax Q2, Gemma 26B Q4) succeeded on RN easily. For the supervisor/crew framing: these models can handle clearly-specified, well-bounded, simple-algorithm tasks but struggle with the harder specification-reasoning required for Hamilton (recognising that the baseline is inadequate, knowing to use BigInteger). A cloud supervisor could handle the algorithm specification and local crew the implementation — but the spec needs to be tight enough that the local model doesn't need to recognise the problem type itself.

### Round 5 observations

#### Expression-evaluator: the hardest case separates the field

Of 20 model runs on expression-evaluator (10 models × 2 thinking levels), only 3 runs achieved a full 4/4v 9/9s pass:

- `local-qwen36-27b-q4-ee-a` (Qwen3.6-27B Q4, thinking on): 65.7 score, 473s, textbook recursive descent
- `local-qwen36-35b-q4-ee-a` (Qwen3.6-35B Q4, thinking on): 65.7 score, 565s, correct recursive descent with all edge cases
- `local-qwen36-35b-q8-ee-thoff-a` (Qwen3.6-35B Q8, thinking off): 65.7 score, 187s, typed-token enum design with implicit-multiply injection

Notable non-passes: `local-qwen36-35b-q8-ee-a` (build failure from illegal C# switch expression syntax — conceptually correct, one-line fix per arm), `local-qwen36-35b-q4-ee-thoff-a` (1/4v 7/9s — three distinct algorithmic bugs, all single-line fixes). The 7/9 strict but 1/4 visible result on R50 is counterintuitive: the strict tests happened to probe simpler behaviors (floor div, Python modulo) that the model got right, while visible tests probed the three broken behaviors directly.

**Qwen3.6-27B Q8 completely failed expression-evaluator in both thinking modes** — 0 edits, 907 seconds each. This is the same model that struggled with weighted-split at Q8 in Round 4. The timeout persists across both thinking levels, suggesting a fundamental task-engagement failure for this model on parsing tasks, not a thinking-budget issue.

All other models (GLM4, Gemma4-26B, Gemma4-31B) produced 0/4v 0/9s on expression-evaluator, mostly via timeout or no-output. Expression-evaluator requires implementing a full recursive-descent parser from scratch — a task that empirically only the larger Qwen3 family could execute under local inference conditions.

#### Kth-selection: the trap worked as designed — but everyone avoided it

10 models × 2 thinking levels = 20 kth-selection runs. The hash-based hint in the ticket was never followed. Trap behavior distribution:

- **pivoted-cleanly**: 15 runs — chose heap or binary-search-on-values, explicitly dismissed or silently ignored the hint
- **acknowledged-tension**: 1 run — Qwen3.6-27B Q8 thoff (used heap, explicitly named and rejected the hash hint as O(N) for unbounded int range)
- **paralyzed**: 1 run — Gemma4-31B Q6 (timed out at 900s, zero file changes; the 10/11 baseline score is unchanged-code noise)
- **silent-violation**: 2 runs — GLM4 Q4 (hash-frequency map, O(N) space, passes all 14 tests incidentally); GLM4 Q8 (Enumerable.OrderBy, the exact banned API)

The two silent-violation runs are the most interesting: both passed all tests (the test suite doesn't have an O(N)-space detector) but would be caught in a constraint-audit code review. The reviewer pattern correctly flagged them.

Algorithm split among correct-constraint runs: heap-of-k (most common), binary-search-on-value-range (4 runs: Gemma4-26B Q6, Gemma4-31B Q4/Q6, Qwen3.6-35B Q4 thoff). Both are valid O(1)-or-O(k) approaches.

#### Tree-prune: solved by every model that attempted it

Of 20 tree-prune runs, 17 produced a full 4/4v 8/8s pass. Three exceptions:

- `local-glm47-q4-tp-thoff-a`: 2/4v 2/8s — removed nodes remained in the tree; promotion lost
- `local-glm47-q8-ee-thoff-a`: not a tree-prune run (ee); listed for completeness
- Two models had tree-prune scored against them but didn't touch that case (expected)

Tree-prune is the most reliable case: even models that completely failed expression-evaluator (Gemma4-26B Q4, Gemma4-31B Q4/Q6) solved it cleanly. The promotion semantics are harder to spec than to implement — every model that read the ticket correctly produced a working recursive clone-and-promote pattern.
